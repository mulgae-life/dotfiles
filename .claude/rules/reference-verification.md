# 레퍼런스 검증 규칙

> 학술 논문·연구 보고서·기술 사양서를 인용하거나, 논문 수식을 코드로 구현하거나, benchmark 수치를 비교할 때 적용.
> 라이브러리/프레임워크 공식 문서는 대상이 아니다.

## MUST — 논문 인용

- **원문 확인 없이 구체적 내용 언급 금지**. "아마 ~일 것이다" 식의 추측 답변 금지
- **Abstract만으로 답변 금지**. 구체적 내용은 해당 섹션 본문까지 읽어야 한다
- **사용자 제공 자료 우선**. 웹 검색은 그 이후에 보충용
- **전문 확보 순서**: 사용자 자료 → ar5iv → WebSearch → Semantic Scholar → PubMed Central → 저자 홈페이지 → HTML 버전
- **접근 실패 시 솔직하게 고백**. 읽지 못한 부분을 추측으로 채우지 않는다

## MUST — 수식↔코드 대조

- **코드가 "Eq.(X) 구현"이라고 주장하면, 항 단위 대조표를 만들어 검증**
  - ✗ "added mass를 effective mass로 치환했으니 Eq.(21)과 같다" (개념 수준 판단)
  - ✓ 논문 Eq.(21) 우변: `F_h + (ρ_p-ρ_f)Vg`, 분모: `ρ_p V`. 코드: `F_total / mass_eff` where `mass_eff = mass_p + m_fluid` → 분모가 다르다. faithful 아님
- **코드 내부에 "heuristic", "exploratory", "project-specific" 라벨이 있으면 faithful로 전제하지 않기**
  - ✗ METHOD_REFERENCE_MAP에 "heuristic"이라고 적혀 있지만 결과가 비슷하니 faithful일 것
  - ✓ "heuristic" 라벨 발견 → 논문 수식과 실제 구현을 항 단위로 대조하여 차이 확인

## MUST — benchmark 수치 비교

- **비교 전에 논문 원문에서 해당 무차원수의 정의 수식을 직접 확인**
  - ✗ Wang 2008 `Re_max = 484.75`를 그대로 우리 `Re_standard`와 비교 → "30% gap" 오진
  - ✓ Wang Eq.(43) 확인 → `Re = ρ_p U D / μ` (particle basis) → 우리 `Re_standard = ρ_f U D / μ`와 다름 → `Re_standard = 484.75 / 1.5 = 323.17`로 환산 후 비교
- **정의가 다르면 환산식을 명시적으로 쓰고 나서 비교**

## MUST — 디버깅 사고

- **내가 구현한 코드는 "틀렸다"를 기본 전제로 시작. 맞다는 증거를 찾기**
  - ✗ 내가 짠 `euler_explicit_step`은 Eq.(21) 기반이니까 맞을 것 → 다른 원인 탐색
  - ✓ `euler_explicit_step`의 분모가 `mass_eff`인데, Eq.(21) 분모는 `ρ_p V`다 → 불일치 발견
- **가설은 확증이 아니라 반증을 먼저 설계**
  - ✗ "DF enforcement 부족" 가설 → MDF로 실험 → 개선됨 → 가설 확인 (확증만 수집)
  - ✓ "DF enforcement 부족" 가설 → "그렇다면 같은 DF인데 이산화만 바꾸면 안 나아야 한다" → explicit-history로 바꿔봄 → 나아짐 → 가설 반박됨
- **수식 오류, 정의 오독 같은 단순 원인을 전부 배제한 뒤에만 "구조적 한계" 결론 허용**
  - ✗ "diffuse IBM의 parasitic drag가 구조적 한계" (수식 대조 안 한 상태에서 결론)
  - ✓ 수식 항 대조 완료 + 정의 환산 완료 + 단순 버그 배제 완료 → 그래도 gap 있으면 그때 구조적 원인 검토
- **새 실험 설계보다 기존 코드/수식 검증이 먼저**
  - ✗ "marker spacing을 바꿔볼까?" (코드 수식이 맞는지도 확인 안 한 상태)
  - ✓ 먼저 현재 구현의 수식 대조 → 문제 없으면 그때 파라미터 탐색
- **같은 축에서 반복 실패하면 실험 멈추고 전제 목록 작성**
  - 현재 전제를 나열하고 각각 "직접 검증했는가 O/X"를 표기. 미검증 전제부터 재확인

## 검증 프로토콜

```
1. 논문/레퍼런스 관련 질문 수신
2. 사용자가 파일/URL 제공했으면 해당 자료부터 읽기
3. 없으면 WebSearch → ar5iv/HTML 확보 → WebFetch로 본문 읽기
4. 해당 섹션을 실제로 읽은 후에만 답변
5. 답변 시 출처 섹션 명시 (예: "Section 3.2에 따르면…")
```

## AVOID

- **기억 기반 인용**: 원문 확인 없이 "이 논문에서는 ~한 방법을 제안했습니다"
- **숫자/수치 날조**: 원문에서 확인한 수치만 언급
- **논문 간 혼동**: 유사한 제목/저자의 다른 논문 내용을 섞는 것
- **존재하지 않는 논문 인용**: 검색 결과로 존재가 확인된 논문만 인용

## 예외

- **일반 상식 수준**은 검증 불필요: "LBM은 Boltzmann 방정식 기반이다" 등
- 사용자가 **"검색 없이 아는 선에서"** 명시적 요청 시 — 단, "검증되지 않은 내용"임을 고지
