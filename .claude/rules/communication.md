# 커뮤니케이션 규칙

## 언어

- 모든 응답과 설명은 **한국어**로 작성
- 코드 주석도 한국어로 작성 (단, 영문 주석이 이미 있는 파일은 일관성 유지)

## 기술 용어

영어 기술 용어는 한국어 설명과 함께 병기:

```
✓ 스트리밍(Streaming)
✓ 컴포넌트(Component)
✓ 미들웨어(Middleware)
✓ 캐싱(Caching)
```

단, 코드 내 변수명/함수명은 영어 그대로 사용.

## 코드 변경 설명

코드를 수정할 때는 반드시 **이유를 상세히 설명**:

```markdown
✓ 좋은 예시:
"이 함수를 분리한 이유는 단일 책임 원칙(SRP)을 지키고,
테스트를 용이하게 하기 위함입니다. 기존 함수는 데이터 페칭과
UI 렌더링을 모두 담당하고 있어 수정 시 side effect가 발생할
가능성이 높았습니다."

✗ 나쁜 예시:
"함수를 분리했습니다."
```

## 에러 설명

에러 발생 시 **원인**과 **해결책**을 함께 제시:

```markdown
## 에러
`TypeError: Cannot read property 'map' of undefined`

## 원인
`fetchData()` 함수가 데이터를 반환하기 전에 컴포넌트가
렌더링되어 `data`가 `undefined` 상태입니다.

## 해결책
1. 옵셔널 체이닝 사용: `data?.map(...)`
2. 초기값 설정: `const [data, setData] = useState([])`
3. 로딩 상태 추가: `if (loading) return <Spinner />`
```

## 작업 제안

작업 제안 시 **구체적인 근거**와 함께 제시:

```markdown
✓ 좋은 예시:
"이 API 엔드포인트에 캐싱을 추가하는 것을 권장합니다.
현재 동일한 데이터를 5초 간격으로 반복 호출하고 있으며,
Redis 캐시를 적용하면 DB 부하를 80% 줄일 수 있습니다."

✗ 나쁜 예시:
"캐싱을 추가하면 좋을 것 같습니다."
```
