# 문체 점검 상세 카탈로그 (한국어 산출물)

> work-verify 4단계 "문체 점검"의 상세 대조표. 규칙의 정답은 `~/dotfiles/.claude/rules/communication.md`이고,
> 이 파일은 점검할 때 참조하는 패턴 카탈로그다.
> 출처(직접 열람분): Wikipedia "Signs of AI writing"(위키피디아 AI 정리팀의 신호 목록),
> 번역투 교정 가이드(egowriting), AI 문체 검출 가이드(리듬 휴리스틱) — 2026-07-13 조사.
> 추가 출처: KatFishNet(ACL 2025)·국립국어원 새국어생활(2012)·토스 테크니컬 라이팅·무하유 GPT킬러 — 상세는 dotfiles 레포의 `reference/research/research-korean-natural-style.md`.
> 번역투 패턴 예문은 위 가이드를 참고해 기술 문서 맥락으로 재작성한 것.

## 빠른 검출 요령 3가지

1. **낭독 테스트**: 소리 내어 읽었을 때 호흡이 끊기거나 입에 붙지 않는 문장을 표시한다.
2. **삭제 테스트**: 문장을 통째로 지워도 정보가 사라지지 않으면 사족이다. 지운다.
3. **리듬 검사**: 모든 문단이 비슷한 길이·구조·강도로 반복되면 AI 신호다. 사람 글은 리듬이 불균질하다 — 중요한 곳은 길고 사소한 곳은 짧다.

## 1. AI스러운 문체 신호

### 1.1 과장·공허 수식 (puffery)

내용 없이 중요성만 부풀리는 표현. 삭제 테스트에 100% 걸린다.

- ✗ "중요한 이정표가 되었습니다", "~의 방증입니다", "핵심적인 역할을 수행합니다", "게임 체인저"
- ✗ "역동적인", "풍부한", "혁신적인", "강력한" (근거 없는 형용사)
- ✓ 무엇이 어떻게 바뀌었는지 사실만 쓴다: "빌드 시간이 12분에서 3분으로 줄었다"

### 1.2 대구법·부정 병렬 남발

"단순한 X가 아니라 Y" 구조는 한 문서에 한 번이면 충분하다. AI 문장은 짧은 글에서도 이 구조를 반복하는 경향이 있다.

- ✗ "이것은 단순한 버그 수정이 아니라 아키텍처의 진화입니다"
- ✗ "X를 넘어 Y로", "X가 아닌 Y", "X라기보다는 Y" 반복
- ✓ 주장을 직접 쓴다: "이 수정으로 재연결 로직이 단순해졌다"

### 1.3 3항 나열 강박 (rule of three)

형용사·명사를 습관적으로 3개씩 나열해 포괄적인 것처럼 보이게 하는 패턴.

- ✗ "안정성, 확장성, 유지보수성을 모두 갖춘", "빠르고, 안전하고, 유연한"
- ✓ 실제로 중요한 한두 가지만 근거와 함께 쓴다

### 1.4 유령 출처 (vague attribution)

출처 없는 권위 귀속. 리포트에서 특히 위험하다(사실 검증 불가).

- ✗ "전문가들은 ~라고 지적합니다", "일각에서는", "업계에서는", "관찰자들에 따르면"
- ✓ 출처를 특정한다: "openai/codex #31814 이슈에 따르면" / 출처가 없으면 내 판단임을 명시

### 1.5 계사 회피

"~이다"로 충분한 자리에 거창한 동사를 돌려 쓰는 패턴.

- ✗ "이 모듈은 인증 계층으로 자리매김합니다", "핵심 허브로 기능합니다"
- ✓ "이 모듈은 인증 계층이다"

### 1.6 메타 서술·요약 강박

글이 글 자신을 설명하는 문장. 독자는 본문을 읽으면 안다.

- ✗ "이 문서에서는 ~를 다룹니다", "주목할 점은 ~라는 것입니다", "결론적으로", "요약하자면" (마지막 문단마다)
- ✗ "~라는 점은 아무리 강조해도 지나치지 않습니다"
- ✓ 바로 본론을 쓴다. 요약이 필요하면 문서 맨 앞에 한 번만

### 1.7 "어려움에도 불구하고 발전" 공식

도전과제를 나열한 뒤 기계적으로 낙관 결론을 붙이는 패턴.

- ✗ "이러한 한계에도 불구하고 이 접근법은 계속 발전하고 있습니다"
- ✓ 한계는 한계대로 끝낸다. 낙관 전망은 근거가 있을 때만

### 1.8 압축체·전보문 (telegraphic prose)

설명하는 산문을 명사구 나열·가운뎃점(·) 연결·괄호 주석 연쇄로 눌러 담은 패턴. 정보 밀도는 높아 보이지만 조사·어미가 빠져 독자가 관계를 스스로 복원해야 한다. 표·제목·파일 목록처럼 압축이 목적인 형식은 예외다.

- ✗ "훅 오탐 정정(멀티라인 미대응)·조항 신설·README 반영"
- ✓ "훅이 멀티라인 명령을 오탐하던 것을 바로잡았고, 관련 조항을 새로 만들어 README에 반영했다"
- ✗ 괄호 보충이 잇달아 읽는 호흡을 끊는 문장
- ✓ 괄호를 풀어 문장을 나누고, 작업 중 만든 코드네임·축약 라벨은 첫 사용 시 풀어 쓴다 (§3 용어 표준성과 연동)

### 1.9 서식 신호 (보조 지표)

서식은 문체·한국어·용어 문제보다 후순위다. 짧은 내용의 개조식 구조화 자체는 문제가 아니며, 아래는 **알맹이 없는 부풀리기**일 때만 지적한다.

- ✗ **볼드 남발** — 거의 모든 핵심어를 기계적으로 굵게
- ✗ "**용어**: 설명" 형태의 콜론 정의 리스트가 문서 전체에 반복
- ✗ 줄표(—) 남용 — 쉼표나 괄호가 자연스러운 자리까지 줄표
- ✗ 이모지를 모든 불릿의 리스트 마커로 기계적으로 반복 사용 — communication.md가 허용하는 상태(✅/⚠️)·섹션 구분(📋)·강조(🎯) 이모지는 해당 없음
- ✗ 헤딩 레벨 건너뜀(##→####), 섹션마다 수평선(---) 삽입
- ✗ 내용 없는 헤더, 같은 말을 항목만 쪼갠 불릿
- ✗ "첫째, … 둘째, … 셋째, …" 기계적 순서 열거가 문서 전체에 반복
- ✓ 서식은 정보 위계를 반영할 때만 — 구조가 내용을 흐리는지가 기준이지, 분량이 기준이 아니다

### 1.10 내용 층위 신호 (표층 문체와 독립)

AI 인상은 어휘·구문만이 아니라 내용에서도 발생한다.

- ✗ 개념·원리·방법론 등 추상적 일반론 중심 서술, 이 프로젝트의 구체 사실 부재
- ✗ 판단 주체가 사라진 서술 — 근거가 있는데도 판단·단정 없이 전부 중립 해설
- ✓ 이 대화·이 저장소의 구체 사실, 실측 수치, 근거 있는 단정으로 서술

## 2. 번역투 (어색한 한국어)

| 패턴 | ✗ 번역투 | ✓ 교정 |
|------|---------|--------|
| 피동·수동태 | "설정이 시스템에 의해 로드되었다" | "시스템이 설정을 로드했다" |
| 이중 피동 | "개선이 이루어지게 되었다", "~되어진다" | "개선했다" |
| 가지다 (have 직역) | "이 함수는 두 개의 역할을 가진다" | "이 함수는 역할이 둘이다" |
| 물주구문 | "이 기능이 사용자로 하여금 ~하게 한다" | "사용자가 이 기능으로 ~할 수 있다" |
| ~로 인해 | "충돌로 인해 빌드가 실패했다" | "충돌 때문에 빌드가 실패했다" |
| ~에 대해/대한 | "에러에 대한 처리가 필요하다" | "에러 처리가 필요하다" |
| ~에도 불구하고 | "재시도에도 불구하고 실패했다" | "재시도했지만 실패했다" |
| ~에 위치하고 있다 | "설정 파일은 루트에 위치하고 있다" | "설정 파일은 루트에 있다" |
| ~를 필요로 하다 (need 직역) | "이 작업은 승인을 필요로 한다" | "이 작업은 승인이 필요하다" |
| ~에 있어(서) | "성능에 있어서 중요한 요소" | "성능에서 중요한 요소" / "성능상 중요한 요소" |
| ~중 하나 (one of 직역) 남발 | "가장 중요한 도구 중 하나" | 근거가 있으면 단정: "핵심 도구다" |
| ~적(的) 남용 | "구조적인 개선적 접근" | "구조 개선 접근" |
| ~들 복수 남용 | "파일들을 읽어 설정들을 적용" | "파일을 읽어 설정을 적용" (한국어는 복수 표지 생략이 기본) |
| 대명사 남용 | "그것은 그것의 상태를 그것에 저장한다" | 명사를 반복하거나 생략: "세션은 상태를 파일에 저장한다" |
| 진행형 직역 | "지원하고 있습니다" (상태 동사) | "지원합니다" |
| 연결어미 뒤 쉼표 남발 | "빌드를 돌리고, 결과를 확인하고, 배포한다" 식 쉼표가 문서 전반에 반복 | 연결 관계를 분명히 할 때만 쓰고 단문 열거는 쉼표 없이 (단발 사용은 정상) |
| 이중 조사 | "환경에서의 실행", "배포로의 전환" | "환경에서 실행", "배포 전환" |
| 명사화·군더더기 한자어 | "삭제하는 작업을 수행합니다", "모니터링 수행을 통해 측정을 실시" | "삭제합니다", "모니터링해서 측정합니다" — 수행하다·실행하다·진행하다는 빼도 의미가 전달된다 |

> 가능형 남발("~할 수 있습니다"), 약화체("~인 것 같습니다"), 상투어 목록은
> `~/dotfiles/.claude/rules/communication.md`의 문체 규칙이 정답 — 여기서는 중복 나열하지 않는다.

## 3. 용어 표준성

- **자의적 용어·내부 코드네임 금지**: 작업 중에 만든 축약어·코드네임·은어를 산출물에 설명 없이 쓰지 않는다. 독자는 작업 과정을 보지 않았다.
  - ✗ "P1-2 프로브 방식으로 재검증했습니다" (P1-2가 뭔지 독자는 모름)
  - ✓ "권한 프롬프트 발생 여부를 임시 파일 삭제 명령으로 재검증했습니다" — 라벨이 꼭 필요하면 첫 사용 시 풀어 쓰고 나서 사용
- **표준 우선순위**: 프로젝트 용어집(있다면) → 업계 관용 표기 → 국립국어원/TTA 표준. 충돌하면 프로젝트 용어집이 이긴다.
- **임의 번역·희귀 한자어 금지**: 정착된 외래어를 창작 번역하지 않고(✗ "흐름 전송" → ✓ "스트리밍"), 사전에 있어도 일상에서 안 쓰는 한자어 압축어를 쓰지 않는다 (✗ "축자 대조"·"오귀속" → ✓ "원문 그대로 대조"·"출처를 잘못 연결").
- **문서 내 일관성**: 같은 개념은 한 표기로 통일 (✗ "구독"/"서브스크립션" 혼용, ✗ "레포"/"저장소"/"리포지토리" 혼용).
- **병기 규칙**: 첫 등장만 한글(영문), 이후 한글만. 변수명·CLI 명령·고유명사는 영어 그대로.
- **외래어 표기 일관성**: 같은 문서에서 표기 흔들림 금지 (✗ "메세지"와 "메시지" 혼용 — 표준은 "메시지").
