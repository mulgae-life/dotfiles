# 피드백 분석 가이드

사용자/서비스로부터 수집된 피드백을 체계적으로 분류하고, 코드 탐색을 통해 근본 원인을 특정하여 우선순위별 개선 리포트를 작성합니다.

## 피드백 유형 분류

모든 피드백을 아래 5가지 유형으로 분류합니다. 하나의 피드백이 복수 유형에 해당하면 가장 심각한 유형을 주(Primary) 유형으로 지정합니다.

| 유형 | 코드 | 설명 | 예시 |
|------|------|------|------|
| 버그 | BUG | 기능 오류, 크래시, 데이터 손실 | "결제 버튼 반응 없음", "저장 후 데이터 사라짐" |
| 사용성 | UX | 사용 흐름 혼란, 직관성 부족 | "설정 메뉴를 찾기 어려움", "뒤로 가기 동작 이상" |
| 기능 요청 | FEATURE | 신규 기능 추가/기존 기능 확장 요청 | "다크 모드 지원해주세요", "엑셀 내보내기" |
| 성능 | PERF | 속도 저하, 응답 지연, 메모리 문제 | "페이지 로딩 5초 이상", "스크롤 버벅임" |
| 콘텐츠 | CONTENT | 텍스트 오류, 디자인 불일치, 번역 문제 | "오타 발견", "아이콘 깨짐", "번역 누락" |

## 분석 방법론

5단계 분석 프로세스를 따릅니다. 단순한 피드백(CONTENT, 명확한 BUG)은 1-2-5 경량 경로로 빠르게 처리합니다.

### 1단계: 피드백 정규화

원문에서 구조화된 정보를 추출합니다.

- **증상**: 사용자가 실제로 겪은 문제 (What)
- **기대 동작**: 사용자가 예상한 동작 (Expected)
- **실제 동작**: 실제로 발생한 결과 (Actual)
- **맥락**: 디바이스, OS, 브라우저, 특정 조건 등

> 원문이 모호한 경우 "추정"으로 표기하고, 재현 확인이 필요함을 명시합니다.

### 2단계: 재현 경로 파악

사용자 행동 경로(User Journey)를 역추적합니다.

- 사용자가 어떤 화면에서 어떤 순서로 조작했는지 파악
- 해당되는 경우: 재현 스텝을 나열 (Steps to Reproduce)
- 조건부 재현: 특정 데이터/상태/타이밍에서만 발생하는지 확인

### 3단계: 코드 탐색

Grep/Glob으로 관련 코드를 탐색하여 원인을 특정합니다.

- **UI 문제**: 관련 컴포넌트 파일, 이벤트 핸들러, 스타일 코드 탐색
- **API 문제**: 라우트 핸들러, 미들웨어, 에러 핸들링 로직 탐색
- **데이터 문제**: DB 쿼리, 스키마, 마이그레이션, 유효성 검증 로직 탐색
- **성능 문제**: N+1 쿼리, 불필요한 리렌더링, 무거운 연산, 캐싱 부재 탐색

> 코드 탐색 결과 영향 파일 목록과 구체적 라인을 기록합니다.

### 4단계: 로그/에러 확인

해당되는 경우, 에러 메시지와 로그에서 근거를 수집합니다.

- 콘솔 에러 메시지, 서버 로그, 에러 트래킹 서비스 데이터
- 에러 핸들링 코드에서 누락된 케이스 확인
- 타임아웃, 네트워크 에러, 유효성 검증 실패 등 패턴 식별

### 5단계: 영향도 평가

아래 기준으로 심각도와 영향 범위를 평가합니다.

- **영향 사용자 수**: 전체 사용자 / 특정 조건 사용자 / 극소수
- **핵심 플로우 관련 여부**: 결제, 인증, 핵심 기능 등 크리티컬 경로인지
- **워크어라운드 존재 여부**: 사용자가 우회할 수 있는 방법이 있는지
- **데이터 손실 위험**: 사용자 데이터 손실/오염 가능성

## 우선순위 기준

### 등급 정의

| 우선순위 | 설명 | 대응 기한 |
|----------|------|-----------|
| P0 | 서비스 이용 불가, 핵심 기능 오류, 데이터 손실 | 즉시 |
| P1 | 주요 기능 사용성 저하, 성능 심각 저하 | 1-2일 |
| P2 | 부가 기능 문제, 경미한 UX 불편 | 1주 내 |
| P3 | 개선 제안, 선택적 기능, 미세 조정 | 백로그 |

### 영향도 매트릭스

심각도와 영향 범위를 교차하여 우선순위를 결정합니다.

|  | 영향 범위 넓음 (전체/다수) | 영향 범위 좁음 (소수/특정 조건) |
|--|---------------------------|-------------------------------|
| **심각도 높음** (기능 불가/데이터 손실) | P0 | P1 |
| **심각도 중간** (사용성 저하/성능 저하) | P1 | P2 |
| **심각도 낮음** (불편/개선 제안) | P2 | P3 |

> 경계 케이스: 핵심 플로우(결제/인증)에서 발생하면 한 단계 상향합니다.

## 워크플로우

피드백 분석 시 아래 체크리스트를 순서대로 따릅니다.

### 분석 단계

- [ ] 1. 피드백 수집 및 유형 분류 (5가지 유형 코드 배정)
- [ ] 2. 유형별 분석 방법론 적용 (단순 피드백은 경량 경로)
- [ ] 3. 영향도 매트릭스로 우선순위 배정
- [ ] 4. 리포트 작성 (`references/report-template.md` 참조)
- [ ] 5. 작업 권장 순서 정리 (P0 → P1 → P2 → P3)

### 분석 시 확인 항목

- [ ] 피드백이 재현 가능한지 확인 (또는 재현 필요 표기)
- [ ] 관련 코드 파일/라인 특정
- [ ] 영향 범위 파악 (다른 기능에 미치는 영향)
- [ ] 해결 방안의 사이드이펙트 검토

### 리포트 작성 후

- [ ] 영향도 매트릭스 기준에 맞게 분류되었는지 확인
- [ ] 모든 피드백에 근거(Evidence) 명시
- [ ] 해결 방안에 구체적 수정 방향 포함
- [ ] 작업 권장 순서가 우선순위 반영

## 리포트 구조

리포트는 `references/report-template.md` 템플릿을 기반으로 작성합니다.

### 저장 위치

- `agent-guide/reviews/` 디렉토리가 있으면: `agent-guide/reviews/feedback_YYMMDD.md`
- 없으면: `.claude/scratch/feedback_YYMMDD.md`

### 각 피드백 항목의 필수 필드

```markdown
#### [FB-001] 피드백 제목
- **유형**: BUG / UX / FEATURE / PERF / CONTENT
- **우선순위**: P0 / P1 / P2 / P3
- **원문**: "사용자 원문 인용"
- **증상**: 기대 동작 vs 실제 동작
- **근거(Evidence)**: 코드 위치, 로그 메시지, 재현 조건 등
- **원인 추정**: 코드/설계 관점 분석
- **해결 방안**: 구체적 수정 제안
- **영향 범위**: 영향받는 사용자/기능 범위
- **영향 파일**: 수정 필요 파일 목록
```

> 근거(Evidence) 필드는 코드 탐색/로그 확인 결과를 반드시 포함합니다. 추정만으로는 P0/P1 배정 불가 — 코드/로그 근거가 없으면 "근거 확보 필요"로 표기합니다.
