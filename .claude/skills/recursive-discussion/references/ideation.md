# Mode 0: Ideation

> `recursive-discussion` 스킬의 아이데이션 모드 상세 가이드.
> 초안이 아예 없고, 문제 정의나 가설 발굴부터 시작할 때 사용합니다.

## 워크플로우

1. **Claude가** 질문을 1문장으로 다시 씁니다.
2. **Claude가** 아이디어를 3~5개까지 발산합니다.
3. **Codex가** 각 아이디어의 장점, 리스크, 검증 가능성을 비판적으로 분석합니다.
4. Claude와 Codex가 반론을 주고받으며 아이디어를 좁힙니다.
5. **Claude가** 살아남은 아이디어를 `실행 가능한 가설` 또는 `문서 구조`로 다시 정리합니다.

목표는 "좋아 보이는 아이디어를 많이 만드는 것"이 아니라,
**버릴 아이디어를 빨리 버리고, 남길 아이디어를 검증 가능한 형태로 만드는 것**입니다.

## Packet 형식

아이데이션용 discussion packet:

```text
---
from: claude
to: codex
round: 1
type: ideation-critique
instruction: respond-only, do-not-call-back
---

문제 정의: [풀려는 질문을 한 문장으로]
후보 가설:
1. [가설 A — 기대 효과와 약점]
2. [가설 B — 기대 효과와 약점]
3. [가설 C — 기대 효과와 약점]
근거:
- [실험 데이터/코드 경로]
- [참고 논문/문헌]
중점 검토:
- 독립 연구거리로 성립하는지
- 단순 파라미터 튜닝에 불과한지
- 일반화 가능성이 있는지
```

> packet 끝에 Reply Contract 형식(`runtime.md` §Reply Contract)을 반드시 포함합니다.

## 토론 상태표

아이데이션에서는 토론 상태표를 아래처럼 씁니다:

| # | 쟁점 | A측 주장 | B측 주장 | 상태 | 조치 |
|---|------|---------|---------|------|------|
| 1 | 파라미터 X는 단순 튜닝인가 | 단순 튜닝일 뿐 | 안정화 메커니즘으로는 성립 | 부분 합의 | framing 수정 |
| 2 | 일반화 가능성 | 일반화 불가 | 조건 의존성이 큼 | 합의 | 범위 제한 명시 |

## 수정 우선순위

1. 가설의 검증 가능성 확보
2. 너무 넓은 주장 범위 축소
3. 반례를 견디는 framing으로 조정
4. 실험 계획 또는 비교 축으로 구체화
5. 버릴 아이디어 제거

## 수렴 축

가능한 한 아래 4개 축으로 수렴시킵니다:

| 축 | 질문 |
|----|------|
| Novelty | 기존에 이미 알려진 이야기인가 |
| Value | 이걸 알면 무엇이 좋아지는가 |
| Testability | 지금 코드/데이터로 검증 가능한가 |
| Scope | 논문/리포트 한 섹션으로 감당 가능한가 |

## 결과 분류

아이데이션 결과는 아래 중 하나로 끝나야 합니다:

- **채택**: 바로 실험/문서화로 이어질 가설
- **보류**: 추가 데이터가 있어야 판단 가능한 가설
- **폐기**: 흥미롭지만 현재 범위 밖인 가설

## 산출물 형식

### 1. 문제 정의
- 지금 풀려는 질문 / 왜 중요한지

### 2. 후보 가설
- 가설별 장점 / 리스크 / 검증 가능성

### 3. 반론 처리 결과
- 각 반론의 토론 상태 (합의/미합의/보류)

### 4. 최종 채택안
- Claude가 최종 통합한, 다음 실험 또는 문서화 대상으로 남길 가설

## 핵심 원칙

- 아이디어 개수를 늘리는 것보다 **실행 가능한 가설과 버릴 가설을 분리하는 것**이 우선.
- 멋있어 보이는 표현만 남기고 검증 계획 없이 끝내지 않습니다.
- Claude는 먼저 확장하고 구조화하고, Codex는 무리한 일반화, 가정 누락을 비판적으로 검증합니다.
- 필요하면 Claude가 Codex 반론에 재반론을 제기해도 됩니다.
