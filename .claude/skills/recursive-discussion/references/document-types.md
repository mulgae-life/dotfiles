# 문서 유형별 적용 포인트

## 논문(Paper)

- novelty가 과장되지 않았는지 확인합니다.
- 주장마다 근거 표/그림/실험이 있는지 점검합니다.
- reviewer가 지적할 만한 표현을 먼저 낮춥니다.

## 기술 리포트(Report)

- 표 숫자와 결론 문장이 정확히 맞는지 점검합니다.
- 해석이 데이터보다 앞서가지 않게 합니다.
- `항상`, `명확히`, `우수` 같은 단정 표현은 데이터가 충분할 때만 씁니다.

## 설계 문서(Design Doc)

- 구현 계획과 실제 코드 구조가 맞는지 확인합니다.
- 아직 없는 API/필드를 있는 것처럼 쓰지 않습니다.
- 리스크 완화책과 fallback을 명시합니다.

## 아키텍처 결정 문서(ADR)

- 선택지(Options)와 각 선택지의 trade-off가 명확한지 점검합니다.
- "왜 이 선택지를 골랐는지"가 아닌 "왜 다른 선택지를 버렸는지"에 더 집중합니다.
- 결정의 되돌림 비용(reversibility)을 명시합니다.

## 기술 제안서(RFC/Proposal)

- 문제 정의가 충분히 구체적인지 확인합니다.
- 제안된 해결책이 문제 범위를 벗어나지 않는지 점검합니다.
- 구현 복잡도 대비 기대 효과가 정당화되는지 검증합니다.
- 마이그레이션/하위 호환성 계획이 포함되었는지 확인합니다.

## 사업 기획서(Business Plan)

- 시장 규모/성장률 주장에 출처가 있는지 확인합니다.
- 경쟁 우위 주장이 구체적 근거(기술 차별점, 데이터)로 뒷받침되는지 점검합니다.
- 재무 추정의 가정(assumption)이 명시되어 있는지 확인합니다.
- "최초", "유일한", "혁신적" 같은 단정 표현을 검증합니다.

## 아이데이션(Ideation)

- Claude는 먼저 확장하고 구조화합니다.
- 양쪽 모두 무리한 일반화, 가정 누락, 검증 가능성을 비판적으로 검증합니다.
- 최종 산출물은 "아이디어 목록"보다 **"실행 가능한 가설 목록"**이 되도록 정리합니다.
- 상세 가이드: `ideation.md` 참조.
