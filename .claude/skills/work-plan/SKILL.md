# 작업 계획서 가이드

복잡한 작업의 실행 계획서를 작성합니다.
Phase별로 구체적인 코드/문서 변경 예시를 포함하여, **컨텍스트 없는 작업자가 계획서만 보고도 작업을 수행할 수 있는 수준**으로 작성합니다.

## 작성 스타일

**개조식(Structured) 원칙** — 산문체(Prose) 금지.

- 모든 내용을 **불릿 포인트, 테이블, 체크리스트, key-value 쌍**으로 작성
- 한 항목 = 한 문장 이내. 부연이 필요하면 하위 불릿으로 분리
- 문단형 서술(3문장 이상 연속된 산문)이 나오면 구조화가 부족한 신호

## 규모 판단

계획서 작성 전 작업 규모를 먼저 판단합니다.

| 규모 | 기준 | 계획서 모드 |
|------|------|-----------|
| **소** | 변경 파일 1-2개, 단일 관심사 | **Lite** — 목표 + 범위 + 작업 목록 + 검증만 (Phase 불필요) |
| **중** | 변경 파일 3-7개, 명확한 단계 | **Standard** — 이 스킬의 기본 형식 |
| **대** | 변경 파일 8개+, 아키텍처 결정 필요 | **Standard + 설계 문서** — 별도 설계 논의 후 계획서 |

> Lite 모드: `## 개요` + `### 성공 기준` + `### 범위` + `## 작업 목록` (체크리스트) + `## 검증` 만으로 구성. 파일명 규칙은 동일.

## 필수 섹션

계획서에 반드시 포함해야 하는 섹션입니다. 하나라도 빠지면 불완전한 계획서입니다.

| 섹션 | 핵심 질문 | 빠지면 생기는 문제 |
|------|----------|-------------------|
| **성공 기준** | "완료"를 어떻게 판단하나? | 작업 끝나도 "이게 맞나?" 반복 |
| **범위 (포함/제외)** | 뭘 하고 뭘 안 하나? | 스코프 크립, 불필요한 작업 |
| **변경 예시** | 구체적으로 뭐가 바뀌나? | 계획과 구현 사이 괴리 |
| **리스크 + 대응** | 뭐가 잘못될 수 있나? | 차단 시 재계획 비용 |
| **정량 요약** | 파일 몇 개, Phase 몇 개? | 작업 규모 오판 |
| **Phase별 검증** | 이 Phase가 잘 됐는지 어떻게 확인? | 오류 누적 후 뒤늦게 발견 |

## Phase 분류

| Phase | 목적 | 설명 |
|-------|------|------|
| **Phase 0** | 사전 준비 | 환경 설정, 의존성 설치, 설정 파일 변경 |
| **Phase 1** | 기반 구축 | 타입 정의, 인터페이스, 공통 유틸리티 |
| **Phase 2** | 핵심 구현 | 비즈니스 로직, 데이터 레이어, API |
| **Phase 3** | 연결 & 통합 | UI 연결, 라우팅, 기존 코드 마이그레이션 |
| **Phase 4** | 검증 & 마무리 | 테스트, 문서화, 정리 |

> Phase 수는 작업 규모에 맞게 조정합니다. 단순한 작업은 2-3 Phase, 대규모 작업은 5+ Phase.

## 계획서 형식

### 파일명 규칙

반드시 아래 형식을 따릅니다. 이 규칙이 있어야 계획서를 찾고 관리할 수 있습니다.

```
agent-guide/plans/work-plan_YYMMDD_{주제-kebab-case}.md
```

예시: `work-plan_260402_install-sh-improve.md`, `work-plan_260402_rules-lazy-load.md`

> `agent-guide/plans/` 디렉토리가 없으면 `.claude/plans/`에 저장합니다.

```markdown
**이 계획에 대해 의견이 있으시면 말씀해 주세요.**
```

## 작성 절차

### 1단계: 요구사항 정리

사용자 요청에서 다음을 추출합니다:
- **무엇을**: 구현할 기능 / 변경할 대상
- **왜**: 배경과 동기
- **제약 조건**: 기술적/시간적/범위적 제약

### 2단계: 코드베이스 분석

```
1. 관련 파일 탐색 (Glob, Grep)
2. 핵심 파일 내용 분석 (Read)
3. 기존 패턴/컨벤션 파악
4. 의존성 그래프 파악 (import/export 추적)
```

### 3단계: Phase 설계

각 Phase를 설계할 때 확인할 사항:

- [ ] **Phase 간 의존성이 단방향인가**: Phase N은 Phase N-1까지만 의존
- [ ] **각 Phase가 독립 검증 가능한가**: Phase 완료 후 빌드/테스트 통과 가능
- [ ] **Phase 크기가 적절한가**: 한 Phase에 파일 5-7개 이내 (초과 시 분리)
- [ ] **위험도 높은 작업이 앞에 있는가**: 불확실성이 큰 작업을 뒤로 미루지 않기

### 4단계: 변경 예시 작성

각 Phase에 **구체적인 코드/문서 예시**를 포함합니다:

- **신규 파일**: 핵심 구조(타입, export, 주요 함수 시그니처)
- **수정 파일**: diff 형식 또는 before/after
- **삭제 파일**: 삭제 이유와 대체 경로
- **설정 파일**: 추가/변경할 키-값

> 전체 구현 코드를 쓰는 것이 아니라, **의도와 구조를 파악할 수 있는 수준**의 예시를 작성합니다.

#### No Placeholder 규칙

계획서에 아래 표현이 있으면 불완전합니다. 구체적인 내용으로 대체해야 합니다.

- `TBD`, `TODO`, `추후 결정`, `나중에 추가`
- `적절히 처리`, `필요에 따라`, `등등`
- `Task N과 유사하게`, `위와 같은 방식으로`
- 코드 블록 안의 `...` (생략 — 핵심 구조는 반드시 작성)

### 5단계: 리스크 & 검증 계획

- 각 리스크에 **대응 방안**을 반드시 포함
- 검증 계획은 **Phase별 체크포인트**와 **최종 검증**으로 구분

### 6단계: 사용자 논의

계획서 초안 작성 후:
1. 계획서 파일 경로를 공유
2. 핵심 결정 사항(선택지가 있는 부분)을 명시적으로 질문
3. 피드백 반영 후 상태를 "확정"으로 변경

## 체크리스트

### 계획서 작성 후 확인

- [ ] **파일명이 `work-plan_YYMMDD_{주제}.md` 형식인가**
- [ ] **성공 기준**이 체크리스트 형식으로 측정 가능한가
- [ ] **범위**에 포함/제외가 명시적으로 구분돼 있는가
- [ ] **정량 요약**이 있는가 (총 Phase 수, 변경 파일 수)
- [ ] 모든 Phase에 변경 예시가 있는가
- [ ] **모든 Phase에 검증 방법**(명령어 + 기대 결과)이 있는가
- [ ] Phase 간 의존성이 명확한가
- [ ] 리스크마다 대응 방안이 있는가
- [ ] **No Placeholder** — TBD/TODO/유사참조/`...` 생략이 없는가
- [ ] 기존 코드 패턴/컨벤션을 따르고 있는가
