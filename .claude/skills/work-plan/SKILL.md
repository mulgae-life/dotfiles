# 작업 계획서 가이드

복잡한 작업의 실행 계획서를 작성한다. Phase별로 **구체적인 코드/문서 변경 예시**를 포함하여, 컨텍스트 없는 작업자가 계획서만 보고도 작업을 수행할 수 있는 수준으로 작성한다.

## 작성 스타일

**개조식(Structured) 원칙** — 산문체(Prose) 금지.

- 불릿 · 테이블 · 체크리스트 · key-value 쌍으로 작성
- 한 항목 = 한 문장 이내. 부연은 하위 불릿으로 분리
- 문단형 서술(3문장 이상 연속) → 구조화 부족 신호

## 분량 가이드

**콤팩트 우선** — 템플릿을 "채우기 위한 채우기" 금지. 해당 없는 선택 섹션은 섹션째 생략 (빈 헤더 남기지 않기).

### 섹션별 상한

| 섹션 | 상한 | 비고 |
|------|------|------|
| TL;DR | Lite 3줄 / Standard·Master 5줄 | 한눈 파악용. master 최상단 필수 |
| 개요 > 목표 | 2문장 | |
| 개요 > 배경 | 3줄 | 선택. 자명하면 생략 |
| 성공 기준 | 체크리스트 3-5개 | 측정 가능하게 |
| 변경 예시 | 핵심 시그니처/구조만 | 전체 구현 X |

### 필수 / 선택 구분

각 템플릿 섹션 헤더에 `(필수)` / `(선택)` 태그가 붙어 있다.

- **필수**: 반드시 작성
- **선택**: 작업에 해당될 때만. 해당 없으면 섹션 자체 생략

### 변경 예시 작성 원칙

- **타입 시그니처 · 주요 함수 시그니처 · 핵심 로직 흐름** 수준으로 작성
- 계획서는 청사진이지 최종 코드가 아님 — 전체 구현 복붙 금지
- 언어 예시는 **프로젝트 언어 1개만**. 다른 언어는 "해당 언어의 스키마 라이브러리 사용" 주석만

## 규모 판단 → 모드 선택

계획서 작성 전 규모를 먼저 판단하고, 해당 모드의 템플릿을 사용한다.

| 규모 | 기준 | 모드 | 템플릿 |
|------|------|------|--------|
| **소** | 파일 1-2개, 단일 관심사 | **Lite** | `references/TEMPLATE.md` |
| **중** | 파일 3-9개, 명확한 단계 | **Standard** | `references/TEMPLATE.md` |
| **대** | 파일 10+ **OR** Phase 4+ **OR** 예상 500줄+ | **Split** | `references/TEMPLATE-master.md` + `references/TEMPLATE-part.md` |

> **Split 트리거 3가지 중 하나라도 해당하면 분할**. 긴 단일 계획서는 작업 중 중간부 요구사항 누락 위험이 크다 (`.claude/rules/context-management.md` 중간 망각 패턴).

## 파일 위치 결정

```
1. 프로젝트 루트에 agent-guide/ 디렉토리 있는가?
   → 있음: agent-guide/plans/
   → 없음: .claude/plans/
2. 디렉토리 없으면 생성
```

### 파일명 규칙

| 모드 | 형식 | 예시 |
|------|------|------|
| Lite / Standard | `work-plan_YYMMDD_{주제}.md` | `work-plan_260423_rules-lazy-load.md` |
| Split | `work-plan_YYMMDD_{주제}/` (디렉토리) | `work-plan_260423_auth-refactor/` |

주제는 kebab-case.

## Split 모드 상세

### 디렉토리 구조

```
work-plan_260423_auth-refactor/
├── master.md                     # 전체 개요 + Phase 맵 + 리스크 + 검증
├── part1-phase1-types.md         # Phase 1 상세
├── part2-phase2-service.md       # Phase 2 상세
├── part3-phase3-api.md           # Phase 3 상세
└── part4-phase4-migration.md     # Phase 4 상세
```

part 파일명: `part{N}-phase{N}-{kebab-case-name}.md`

### master / part 역할 분담

| 영역 | master | part |
|------|:------:|:----:|
| 목표 · 배경 · 성공 기준 · 범위 | ✅ | — |
| 현황 분석 (관련 파일, 의존성) | ✅ | — |
| 정량 요약 · Phase 맵 | ✅ | — |
| 전체 리스크 · 전체 검증 계획 | ✅ | — |
| Phase 목표 · 작업 목록 · 변경 예시 | — | ✅ |
| Phase별 검증 · 완료 기준 | — | ✅ |
| 선행 Part 전제 조건 · 후속 Part 참조 | — | ✅ |

### Phase 맵 (master 필수 섹션)

master.md에는 다음 형식의 Phase 맵이 반드시 포함된다.

| Phase | Part | 범위 | 파일 수 | 상태 |
|-------|------|------|---------|------|
| 1 | [part1](part1-phase1-types.md) | 타입 정의 | 3 | ⬜ 대기 |
| 2 | [part2](part2-phase2-service.md) | 서비스 레이어 | 5 | 🟡 진행중 |

> 상태: ⬜ 대기 / 🟡 진행중 / ✅ 완료 / ⚠️ 차단
> Phase 완료 시 이 테이블의 상태를 반드시 갱신한다.

### 작업 시 읽기 전략

- **master.md**: 항상 로드 (전체 맥락 유지)
- **part{N}.md**: 현재 진행 중인 Phase만 로드
- 다른 part는 선행/후속 참조 필요 시에만 읽기

### 일관성 유지

- 각 part 상단에 `> master: [../master.md]` 역참조 필수
- master Phase 맵의 파일 링크와 실제 part 파일 존재 여부 일치 필수
- verifier 점검 시 master ↔ part 정합성 확인

## Phase 분류

| Phase | 목적 | 예시 |
|-------|------|------|
| **Phase 0** | 사전 준비 | 환경 설정, 의존성 설치 |
| **Phase 1** | 기반 구축 | 타입 정의, 인터페이스, 공통 유틸 |
| **Phase 2** | 핵심 구현 | 비즈니스 로직, 데이터 레이어, API |
| **Phase 3** | 연결 & 통합 | UI 연결, 라우팅, 마이그레이션 |
| **Phase 4** | 검증 & 마무리 | 테스트, 문서화, 정리 |

> Phase 수는 작업 규모에 맞게 조정. 단순 작업 2-3 Phase, 대규모 5+ Phase.

## No Placeholder 규칙

계획서에 다음 표현이 남아있으면 불완전하다.

| ✗ 금지 | ✓ 대체 |
|--------|--------|
| `TBD`, `TODO`, `추후 결정`, `나중에 추가` | 구체적 내용 또는 "이 계획서 범위 밖"으로 명시 |
| `적절히 처리`, `필요에 따라`, `등등` | 구체적 조건/대상 명시 |
| `Task N과 유사하게`, `위와 같은 방식으로` | 실제 내용 반복 작성 (독립 이해 가능) |
| 코드 블록 안 `...` (생략) | 핵심 구조(타입/시그니처/주요 로직)는 반드시 작성 |

## 작성 절차

1. **요구사항 정리**: 무엇을 / 왜 / 제약 조건
2. **코드베이스 분석**: Glob/Grep → Read → 기존 패턴·컨벤션 파악
3. **규모 판단 → 모드 선택**: 위 표 기준으로 Lite/Standard/Split 결정
4. **(Split 한정) 분할 설계**: Phase 경계 확정 → Part 매핑 → Phase 맵 초안
5. **템플릿 채우기**: 해당 모드 템플릿을 `references/`에서 읽어 작성 (변경 예시는 프로젝트 언어에 맞게)
6. **리스크 & 검증**: 각 리스크에 대응 방안 필수. Phase별 검증 + 최종 검증 구분
7. **사용자 논의**: 계획서 경로 공유 → 선택지 있는 부분 명시 질문 → 피드백 반영 → 상태 "확정"

### Phase 설계 체크 (Split 모드 포함)

- [ ] Phase 간 의존성이 단방향인가 (Phase N은 Phase N-1까지만 의존)
- [ ] 각 Phase가 독립 검증 가능한가 (Phase 완료 후 빌드/테스트 통과)
- [ ] Phase 크기가 적절한가 (한 Phase에 파일 5-7개 이내)
- [ ] 위험도 높은 작업이 앞에 있는가 (불확실성을 뒤로 미루지 않기)

## 생명주기

| 단계 | 상태 | 동작 |
|------|------|------|
| 작성 | 초안 | 사용자 논의 전 |
| 논의 | 확정 | 사용자 승인 후 |
| 실행 | 진행중 | 구현 시작 (Split은 Phase 맵 상태 갱신) |
| 검증 | 완료 | verifier 에이전트 또는 `/work-verify`로 점검 통과 |
| 정리 | 아카이브 | `.archive/` 하위로 `mv` (저장 위치 기준 매핑) |

### 아카이브 경로 매핑

저장 위치에 따라 아카이브 대상 경로가 달라진다. `rm` 금지, `mv`만 허용.

| 저장 위치 | 아카이브 경로 |
|-----------|--------------|
| `agent-guide/plans/{파일 또는 디렉토리}` | `agent-guide/.archive/` |
| `.claude/plans/{파일 또는 디렉토리}` | `.claude/.archive/` |

> `.claude/rules/agents.md` 규칙: 구현 완료 후 계획서(Split은 디렉토리째)를 위 매핑대로 `.archive/`로 이동.

## 체크리스트

### 공통 (모든 모드)

- [ ] **TL;DR**이 최상단에 있고 줄수 상한(Lite 3줄 / Standard·Master 5줄) 준수하는가
- [ ] **선택 섹션**은 해당되지 않으면 섹션째 생략했는가 (빈 헤더 X)
- [ ] 파일명/디렉토리명이 규칙을 따르는가
- [ ] 성공 기준이 측정 가능한 체크리스트 형식(3-5개)인가
- [ ] 범위에 포함/제외가 명시적인가
- [ ] 정량 요약(Phase 수, 파일 수)이 있는가
- [ ] 모든 Phase(또는 작업 항목)에 **변경 예시 + 검증 방법**이 있는가
- [ ] **변경 예시**가 핵심 시그니처/구조 수준인가 (전체 구현 복붙 X)
- [ ] 리스크가 있다면 각각에 대응 방안이 있는가
- [ ] **No Placeholder** — TBD/유사참조/`...` 생략 없는가
- [ ] 기존 코드 패턴/컨벤션과 일관되는가

### Split 모드 추가

- [ ] master에 **Phase 맵(상태 포함)**이 있는가
- [ ] 각 part가 `> master: [../master.md]`로 역참조하는가
- [ ] master Phase 맵의 파일 링크와 실제 part 파일이 모두 존재하는가
- [ ] 각 part에 **전제 조건**과 **완료 기준**이 있는가

## vs planner 에이전트 / 전체 워크플로우

| planner (자동 위임 에이전트) | /work-plan (수동 호출 스킬) |
|------------------------------|-----------------------------|
| 컨텍스트 내 초안 (응답으로) | 파일로 저장되는 계획서 |
| Step 목록 수준 | Phase별 코드/문서 예시 포함 |
| 빠른 방향 합의용 | 실행 가능한 상세 계획서 |
| 파일 3개+ 수정 시 자동 | 사용자가 `/work-plan` 또는 "계획서 작성해줘" 호출 |

**일반 흐름**: planner 초안 → 사용자 논의 → (복잡하면) `/work-plan`으로 상세화 → 구현 → verifier 점검 → `.archive/` 이동.
