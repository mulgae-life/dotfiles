## 들어가며

개발의 병목이 바뀌었다.

과거에는 "이걸 어떻게 구현하지?"가 문제였다. 이제 AI가 그 과정을 수초 만에 해낸다. 지금의 병목은 **"무엇을 만들어야 하는지 정확히 아는 것"**이다.

AI는 빠르다. **잘못된 방향으로도 빠르다.** 대충 지시하면, 그럴듯하지만 뼈대부터 틀어진 코드를 수백 줄 만들어낸다. 그걸 되돌리느라 직접 짜는 것보다 더 오래 걸린다.

**당신이 사수고, AI가 주니어다.** 10년 차의 구현력과 1년 차의 판단력을 동시에 가지고 있다. 시키는 건 기가 막히게 해내지만, 뭘 시켜야 하는지는 모른다. 사수가 지시를 대충 하면 주니어의 결과물도 대충이 되는 건 똑같다.

이 가이드는 프롬프트를 잘 쓰는 요령이 아니라, **AI와 함께 일하는 개발 프로세스 자체를 세팅하는 방법**을 다룬다.

### AI는 왜 실수하는가

AI가 엉뚱한 코드를 짜는 건 멍청해서가 아니라, 작동 방식이 그렇기 때문이다.

1. **모호하면 추측한다.** "로그인 만들어줘"라고 하면, 프레임워크·인증 방식·에러 처리를 전부 AI가 결정한다. "불명확한데 어떻게 할까요?"라고 묻지 않는다.
2. **보이는 범위만 본다.** 파일 하나만 주면 그 파일만 보고 수정한다. 다른 10개 파일이 깨지는지는 모른다. "전체를 먼저 파악해"라고 한 마디 추가하면 영향도를 고려하기 시작한다.
3. **"모른다"고 말하지 않는다.** 존재하지 않는 함수를 자신 있게 호출하고, 문법이 깔끔해서 틀린 줄 모르고 넘어가기 쉽다.

이 가이드의 원칙들은 이 세 가지에 대한 대응이다. 명세서 → 추측 여지 제거. 전체 파악 지시 → 사이드이펙트 감소. 검증 → 자신 있는 오답 걸러내기.

## 1장. 핵심 원칙

### 원칙 1: Git 없이는 시작도 하지 마라

AI는 수초 만에 파일 10개를 동시에 수정한다. 방향이 맞으면 생산성 폭발이지만, 방향이 틀리면 깔끔하던 코드가 순식간에 뒤엉킨다.

**버전 관리 없이 AI에게 작업을 맡기는 건, 세이브 없이 보스전에 들어가는 것과 같다.**

**반드시 지킬 것:**
- 작업 시작 전 **새 브랜치 생성** — `git checkout -b feature/...`
- 의미 단위마다 **커밋** — AI에게 커밋을 시켜도 된다
- `git diff`를 **습관적으로 확인** — AI가 뭘 바꿨는지 내 눈으로 확인
- 잘못됐다 싶으면 **미련 없이 되돌리기** — 그래서 브랜치가 필요하다

**안 하면?** AI가 기존 로직을 조용히 덮어쓰고, 돌이킬 수 없는 상태에서 수동 역추적에 반나절을 날린다.

---

### 원칙 2: 대화하지 말고, 문서를 밀어넣어라

가장 흔하고 가장 위험한 실수: AI와 채팅으로 핑퐁하면서 요구사항을 조금씩 추가하는 것.

> "로그인 기능 만들어줘" → "비밀번호 틀리면 에러 보여줘" → "5회 초과하면 잠금도" → "이메일 인증도 추가해줘"

왜 위험한가? 첫 지시 "로그인 만들어줘"에서 AI는 이미 자기 나름의 인증 구조를 설계하고 코드를 짠다. 그 다음 "잠금 기능도"라고 하면, 처음 짠 구조와 맞지 않더라도 기존 코드를 뜯어고치는 대신 **위에 덧붙인다.** 요청이 누적될수록 코드는 누더기가 되고, 어디서부터 꼬였는지 추적이 안 된다.

**올바른 방법:** 명세서(`spec.md`, `task.md` 등)에 인증 플로우, 에러 케이스, 잠금 정책, 이메일 인증까지 **한 문서에 정의**해서 한 번에 밀어넣는다. AI는 전체 그림을 보고 나서 구조를 설계하므로, 기능 간 충돌이 처음부터 없다.

```
이 문서를 읽고, 이대로 구현해줘.
시작 전에 네가 이해한 내용을 요약해서 보여줘.
```

30분 투자해서 쓴 명세서 한 장이, 2시간짜리 채팅 핑퐁을 대체한다.

---

### 원칙 3: 구조를 먼저 잡아라

AI는 **구현에는 탁월하지만, 설계에는 취약하다.**

AI는 일반적으로 좋은 패턴은 알지만, **당신의 프로젝트에서 어떤 패턴을 쓰고 있는지**는 모른다. 기존에 서비스 레이어 패턴을 쓰고 있는데 AI가 갑자기 MVC로 새 기능을 만들면, 코드베이스에 두 가지 패턴이 공존한다. 기능은 돌아가지만 유지보수가 지옥이 된다.

**먼저 정해야 할 것:**
- **폴더 구조** — 어디에 뭘 넣을지
- **모듈 경계** — 어떤 파일이 어떤 책임을 가지는지
- **데이터 흐름** — 데이터가 어디서 시작해서 어디로 가는지
- **기존 패턴** — 프로젝트에서 이미 쓰고 있는 방식은?

빈 파일이라도 뼈대를 잡아두면, AI는 그 구조를 존중하며 살을 붙인다. 백지에서 시작하면 AI가 만든 구조를 나중에 리팩토링해야 하는 이중 작업이 생긴다.

---

### 원칙 4: 맥락을 충분히 줘라

**AI의 출력 품질은 입력 맥락의 품질에 정비례한다.**

특히 중요한 건 **"전체 코드를 먼저 파악해"라고 지시하는 것**이다. 이 한 마디가 결과물의 품질을 극적으로 바꾼다.

| | 지시 없이 바로 작업 | "전체를 먼저 파악해" 후 작업 |
|---|---|---|
| **AI가 보는 범위** | 해당 파일만 | 폴더 구조, 의존 관계, 기존 유틸리티 전체 |
| **중복 코드** | 이미 있는 함수를 다시 만듦 | 기존 함수를 찾아서 재사용 |
| **사이드이펙트** | 다른 파일이 깨지는지 모름 | 영향도를 고려해서 수정 |
| **결과** | 에러 많음, 코드 불일치 | **같은 작업인데 에러 확연히 감소** |

**맥락을 주는 방법:**
1. **"프로젝트 전체 구조를 먼저 파악해"** — 작업 전 반드시 지시
2. **프로젝트 규칙 파일** — `CLAUDE.md`, `.cursorrules` 등으로 컨벤션 자동 주입
3. **기존 코드 참조** — "src/services/auth.service.ts의 패턴을 따라서 구현해"
4. **제약 조건 명시** — "기존 API 응답 포맷을 바꾸지 마"

---

### 원칙 5: AI의 출력을 맹신하지 마라

AI는 자신감 넘치는 어조로 틀린 코드를 만든다. 문법은 완벽하고, 변수명도 그럴듯하고, 주석까지 친절하다. 하지만 비즈니스 로직이 미묘하게 틀리거나, 존재하지 않는 함수를 호출하거나, 보안 취약점을 심기도 한다.

위험한 이유: 사람이 짠 버그는 "아, 여기 실수했네" 하고 금방 찾지만, AI가 만든 버그는 **코드가 너무 깔끔해서 틀린 줄 모르고 넘어간다.** "AI가 짰으니까 괜찮겠지"라는 방심이 생기면 문제가 프로덕션까지 간다.

**AI가 짠 코드를 리뷰 없이 커밋하는 순간, 그건 당신의 코드다. 책임은 당신에게 있다.**

**검증 습관:**
- `git diff`로 변경 사항 **전수 리뷰**
- 테스트 실행으로 **동작 검증**
- 복잡한 로직은 AI에게 설명 요청 — "이 함수가 왜 이렇게 동작하는지 설명해"
- 찝찝하면 다시 짜게 하거나, 직접 수정

## 2장. 실전 워크플로우

하나의 기능을 구현하는 전체 흐름.

```
[설계 & 명세 40%] → [환경 준비 10%] → [에이전트 투입 30%] → [검증 20%]
```

### Phase 1: 설계 & 명세 (40%)

**가장 중요한 단계.** 여기에 투자한 시간이 뒤의 모든 시간을 절약한다.

| 항목 | 내용 |
|------|------|
| **목표** | 이 작업이 끝나면 뭐가 달라지는가? 한 문장으로 |
| **상세 요구사항** | 구체적인 기능 목록, API 스펙, UI 동작 |
| **제약 조건** | 기존 시스템 호환성, 기술 스택, 성능 요구 |
| **예외 케이스** | 잘못된 입력, 네트워크 실패, 권한 없음 |
| **참조** | 기존 코드 중 따라야 할 패턴 |

> 명세서 작성 자체에도 AI를 활용할 수 있다. "명세서로 정리해줘"라고 하면 놓친 예외 케이스까지 챙겨준다. 단, 최종 검토와 확정은 반드시 사람이.

### Phase 2: 환경 준비 (10%)

```bash
# 1. 브랜치 생성
git checkout -b feature/board-api

# 2. 필요하면 디렉토리/파일 뼈대 생성
mkdir -p src/features/board
touch src/features/board/{router,service,schema,model}.py

# 3. 프로젝트 규칙 파일 확인
cat CLAUDE.md  # 또는 .cursorrules
```

### Phase 3: 에이전트 투입 (30%)

| 단계 | 지시 예시 |
|------|-----------|
| **1. 맥락 주입** | "프로젝트 전체 구조와 기존 코드 패턴을 먼저 파악해" |
| **2. 명세 전달** | "spec.md를 읽고, 구현 계획을 먼저 보여줘. 코드 짜기 전에 내가 확인할게" |
| **3. 계획 승인 → 구현** | AI의 계획을 리뷰하고 조정한 뒤 진행 |
| **4. 중간 점검** | `git diff`를 수시로 확인하며 방향 이탈 감시 |

### Phase 4: 검증 & 마무리 (20%)

1. 전체 `git diff` 리뷰 — 의도하지 않은 변경이 없는지
2. 테스트 실행 — 기존 테스트가 깨진 건 없는지
3. 실제 동작 확인 — 눈으로 직접
4. 커밋 & PR

## 3장. 명세서 작성법

과거에는 코드가 최종 산출물이었다. 이제는 **명세서가 코드의 원천**이다.

### 좋은 명세서의 4가지 조건

| 조건 | 설명 |
|------|------|
| **명확한 목표** | "무엇을" 만드는지 한 문장으로 요약 가능 |
| **명확한 경계** | "무엇을 하지 않는지"까지 명시 |
| **충분한 디테일** | 입력/출력, 상태, 에러 케이스가 구체적 |
| **참조점** | 기존 코드의 어떤 패턴을 따를지 지정 |

### 나쁜 명세서 → AI가 추측한다

```
게시판 기능 만들어줘.
CRUD 되게 하고, 페이지네이션도 넣어줘.
```

어떤 프레임워크? 페이지네이션 방식은? 인증은? 에러 처리는? 전부 AI 마음대로.

### 좋은 명세서 → AI가 실행한다

```markdown
## 게시판 API 구현

### 목표
사용자가 글을 작성/조회/수정/삭제할 수 있는 RESTful API

### 기술 스택
- FastAPI + SQLModel (기존 프로젝트 패턴)
- Supabase PostgreSQL

### API 엔드포인트
- POST   /api/posts     — 글 작성 (title, content, author_id 필수)
- GET    /api/posts     — 글 목록 (커서 기반 페이지네이션, 20개/페이지)
- GET    /api/posts/:id — 글 상세
- PUT    /api/posts/:id — 글 수정 (작성자만 가능)
- DELETE /api/posts/:id — 글 삭제 (작성자만 가능)

### 제약 조건
- 인증: 기존 auth middleware 사용 (src/middleware/auth.py 참조)
- 에러 응답: 기존 포맷 따름 (src/schemas/error.py 참조)
- 입력 검증: title 1~100자, content 1~10,000자

### 예외 케이스
- 존재하지 않는 글 접근 → 404
- 권한 없는 수정/삭제 → 403
- 유효하지 않은 입력 → 422 (Pydantic validation)

### 하지 않을 것
- 댓글 기능 (별도 태스크)
- 파일 첨부 (다음 스프린트)
```

차이는 명백하다. 명세서를 처음부터 혼자 쓸 필요도 없다:

```
우리 프로젝트에 게시판 기능을 추가하려고 해.
기존 프로젝트 구조를 보고, 구현 명세서를 작성해줘.
API 스펙, 예외 케이스, 제약 조건까지 포함해서.
```

AI가 초안을 만들면 리뷰하고 보완해서 확정한다.

## 4장. 도구별 활용 가이드

도구마다 강점이 다르다. 모든 작업에 하나의 도구만 쓰는 건 망치로 나사를 박는 것과 같다.

| 도구 | 특성 | 강점 | 적합한 작업 |
|------|------|------|-------------|
| **Claude Code** | 터미널 자율 에이전트 | 멀티파일, 프로젝트 전체 맥락 파악, `CLAUDE.md` 규칙, Skills/Hooks 자동화 | 기능 구현, 버그 추적, 리팩토링 |
| **Cursor** | IDE 통합 | 파일 단위 빠른 수정, 실시간 자동완성, `.cursorrules` 컨벤션 | UI 수정, 스타일 변경, 프로토타이핑 |
| **OpenAI Codex** | 자율 에이전트, 샌드박스 | 태스크 통째로 위임, 병렬 실행 | 독립 기능, 테스트 작성 |

하나만 고를 필요 없다. 구조 설계는 Claude Code, 세부 UI는 Cursor, 독립 유틸리티는 Codex — 상황에 따라 조합하면 된다.

## 5장. 에이전트 환경 세팅

매번 "한국어로 답해", "에러 처리는 이렇게" 반복하는 건 시간 낭비다. **한 번 세팅하면 매 세션 자동 적용되는 환경**을 만들어야 한다.

### 프로젝트 규칙 파일

대부분의 AI 코딩 도구는 프로젝트 루트의 "규칙 파일"을 지원한다. 여기에 컨벤션을 적어두면 매 세션 자동 적용된다.

| 도구 | 파일 |
|------|------|
| Claude Code | `CLAUDE.md`, `.claude/rules/*.md` |
| Cursor | `.cursorrules` |
| Codex | `AGENTS.md` |

**담을 것:** 코딩 컨벤션 · 프로젝트 아키텍처 · 기술 스택 · 응답 규칙(언어, 스타일)

이 파일이 있으면 AI는 첫 세션에서도 기존 멤버처럼 일관된 코드를 만든다.

### 프로젝트 맥락 문서

새 프로젝트 시작 시, AI와 기획 대화를 나눈 뒤 맥락을 문서로 정리해두면 된다. 형식은 자유지만 다음 3가지를 갖추면 AI가 프로젝트를 빠르게 이해한다:

- **프로젝트 개요** — 목적, 기술 스택, 아키텍처 요약
- **작업 원칙** — 이 프로젝트만의 규칙, 특수 용어, 주의사항
- **현재 상태** — 진행 중인 작업, 다음 할 일, 결정 사항

이 문서들을 규칙 파일에서 참조하게 하면, 다음 날 이어서 작업할 때 AI가 **어제 맥락을 바로 이어받는다.**

## 참고: dotfiles 레포

> 아래는 이 가이드의 원칙을 바탕으로 **개인적으로 구성한 에이전트 환경** 예시다. 공식 도구가 아니라 하나의 참고 사례이며, 본인의 필요에 맞게 자신만의 환경을 구축하면 된다.

### 왜 만들었나

5장의 규칙 파일과 맥락 문서를 프로젝트마다 매번 새로 만들면? "코딩 스타일은 이렇게", "에러 처리는 이렇게", "한국어로 답해" 같은 규칙을 반복 작성해야 한다. 프로젝트 3~4개를 동시에 돌리면 이것만으로도 상당한 시간이 든다.

이 레포는 **프로젝트 공통 규칙과 도구를 한 곳에 모아, 어떤 프로젝트에서든 자동 적용되게** 만든 환경이다.

- **전역 규칙(Rules)** — 프로젝트 무관하게 항상 적용할 규칙. 한 번 정의 → 모든 프로젝트에서 자동 로드
- **공통 스킬(Skills)** — `/code-review`, `/frontend-design` 같은 반복 작업을 커맨드 하나로 실행
- **프로젝트 초기화** — `/init-project`로 각 프로젝트에 맞는 가이드 문서 자동 생성. 전역 규칙은 공통, **프로젝트 고유 맥락은 이 문서들이 담당**

"프로젝트마다 반복하는 건 전역으로 통합, 프로젝트별로 달라야 하는 건 개별 생성"하는 구조다.

### 시작하기

[dotfiles 레포](https://github.com/mulgae-life/dotfiles)

```bash
git clone https://github.com/mulgae-life/dotfiles.git ~/dotfiles
~/dotfiles/install.sh --dry-run  # 미리보기
~/dotfiles/install.sh            # 설치 (심볼릭 링크 방식)
```

### 구성 요소

| 구분 | 설명 | 해결하는 문제 |
|------|------|---------------|
| **7개 전역 규칙** | 코딩 스타일, 보안, 아키텍처, 커뮤니케이션 | 같은 규칙 반복 설정 |
| **17개 전문 스킬** | `/code-review`, `/frontend-design`, `/init-project` 등 | 복잡한 프롬프트 반복 작성 |
| **자동 위임 에이전트** | 빌드 에러 → `build-resolver`, 보안 코드 → `security-reviewer` | 에러 시 수동 개입 |
| **흐름 유지 훅** | 위험 명령만 확인, 나머지 자동 승인 | AI가 권한 요청마다 멈춤 |
| **프로젝트 초기화** | 가이드 3종(GUIDE/PROJECT/SESSION.md) 자동 생성 | 맥락 문서 작성 부담 |

이 환경을 그대로 쓸 필요는 없다. 중요한 건 원칙이다: **AI에게 반복 설명하는 것이 있다면, 그건 파일로 만들어 자동화할 수 있다.**

## 부록

### A. 안티패턴 모음

| 이렇게 하면 | 이렇게 된다 | 이렇게 해야 한다 |
|---|---|---|
| Git 없이 AI에게 대규모 수정 | 롤백 불가, 수동 복구에 반나절 | 브랜치 생성 후 작업 |
| 채팅으로 요구사항 조금씩 추가 | 앞 지시를 잊고 일관성 깨짐 | 명세서 한 번에 밀어넣기 |
| "알아서 구조 잡아줘" | 프로젝트와 안 맞는 구조 생성 | 구조 먼저, 구현은 그 다음 |
| AI 출력 리뷰 없이 커밋 | 미묘한 버그, 보안 구멍 누적 | diff 리뷰 + 테스트 필수 |
| 하나의 프롬프트에 모든 걸 | 품질 저하, 요구사항 누락 | 논리적 단위로 분리 |
| "전체 코드 고쳐줘" | 건드리면 안 되는 파일까지 수정 | 수정 범위를 명확히 지정 |
| 구조 설명 없이 투입 | 기존 패턴과 동떨어진 코드 | "기존 패턴을 먼저 파악해" 선행 |

### B. 에이전트 투입 전 체크리스트

- [ ] Git 브랜치를 생성했는가?
- [ ] 명세서/요구사항 문서를 작성했는가?
- [ ] 프로젝트 구조와 컨벤션을 AI에게 전달했는가?
- [ ] "기존 코드를 먼저 파악해"라고 지시했는가?
- [ ] AI의 구현 계획을 리뷰하고 승인했는가?
- [ ] 수정 범위를 명확히 지정했는가?

### C. 명세서 템플릿

```markdown
## [기능명]

### 목표
[이 기능이 완성되면 사용자가 할 수 있는 것 — 한 문장]

### 기술 스택
[사용할 프레임워크, 라이브러리, 기존 프로젝트와의 관계]

### 상세 요구사항
[구체적인 기능 목록, API 스펙, UI 동작 정의]

### 제약 조건
[기존 시스템 호환성, 성능 요구사항, 기술적 제약]

### 예외 케이스
[에러 상황별 예상 동작]

### 참조
[기존 코드 중 따라야 할 패턴이나 파일 경로]

### 하지 않을 것
[이번 작업 범위에서 명시적으로 제외하는 것]
```
