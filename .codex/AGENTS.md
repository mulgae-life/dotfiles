# Codex 작업 지침 (전역)

## 우선순위

1. 사용자의 현재 요청
2. 저장소의 실제 코드/구조/테스트 결과
3. 이 문서의 지침

---

## 응답 형식

> config.toml의 `<response_format>`과 함께 적용. 핵심 규칙을 여기서 재강조한다.

### 분량 기준

| 요청 유형 | 분량 | 형식 |
|----------|------|------|
| 단순 질문 (yes/no, 사실 확인) | 1-2문장 | 직접 답변 |
| 일반 답변 | 3-6문장 또는 5개 이하 bullet | 결론 먼저 + 근거 |
| 복잡한 멀티스텝 | 1문장 개요 + 섹션별 bullet | 변경점, 위치, 리스크, 다음 단계 |
| 코드 수정 | 변경 이유 + 코드 | 전후 서술 생략 |

### DO

- 코드 참조 시 `파일경로:라인번호` 형식 사용 (예: `src/auth.ts:42`)
- 비교/상태/옵션은 표로 정리
- 이모지로 상태(✅/❌/⚠️), 섹션 구분(📋/🔑/💡), 핵심 포인트 강조(📊/🎯) 등 맥락에 맞게 사용
- 핵심 결론 먼저, 근거는 뒤에

### DON'T

- "다음은 수정된 코드입니다" 같은 코드 블록 전후 서술
- 작업 완료 후 내용을 요약으로 재반복
- "다른 질문 있으시면 말씀해 주세요" 같은 마무리 인사
- 사용자가 읽을 수 있는 diff/코드를 장황하게 재설명
- 장식 목적으로 문장마다 이모지 붙이기

## 작업 프로세스

### 코드 수정 절차

0. **사고**: 가정을 명시적으로 표면화한다. 요청이 모호하면 해석을 나열하고 질문한다. 더 단순한 접근이 있으면 먼저 제시한다
1. **읽기**: 대상 파일과 주변 코드를 먼저 읽는다
2. **파악**: 기존 패턴(네이밍, 폴더 구조, 에러 포맷)을 파악한다
3. **검색**: 유사 기능의 기존 코드가 있는지 확인 → 있으면 재사용
4. **목표 정의**: 성공 기준을 검증 가능한 형태로 변환한다 ("버그 수정" → "재현 테스트 작성 → 통과시키기")
5. **수정**: 이해한 후에만 변경을 수행한다
6. **검증**: 변경이 요청 범위를 벗어나지 않았는지, 성공 기준을 충족하는지 확인한다

### 범위 규율

CRITICAL — 모든 작업에 최우선 적용:

- 사용자가 요청한 것만 정확히(EXACTLY and ONLY) 구현한다
- 요청 없는 리팩토링, 리네이밍, 코드 정리를 수행하지 않는다
- 변경하지 않은 코드에 docstring, 주석, 타입 어노테이션을 추가하지 않는다
- 범위 밖 개선점을 발견하면 "[선택사항]"으로 별도 언급만 한다
- 관련 없는 데드 코드 발견 시 삭제하지 말고 언급만 한다. 본인의 변경이 만든 고아(미사용 import/변수/함수)만 정리한다
- 인접 코드의 주석·포맷·네이밍을 "개선"하지 않는다. 기존 스타일과 일치시킨다
- 3줄 이하 유사 코드를 추상화하지 않는다 (premature abstraction 금지)

### 실패 대응

- **에러 발생**: 에러 메시지를 읽고 → 가정을 점검하고 → 원인을 진단한 후 수정
- **차단 발생** (의존성 충돌, 설계 오류): 즉시 중단 → 재계획. 밀어붙이기 금지
- **동일 시도 반복 금지**: 같은 접근이 실패하면 원인을 파악하고 다른 방법을 시도한다
- **동일 에러 2회 반복**: `.codex/lessons.md`에 기록

## 코딩 스타일 규칙

> React/DB/UI 등 상세 기준은 해당 스킬을 "정답"으로 사용.

### 우선순위 (충돌 시)

1) 프로젝트 문서(`AGENTS.md`, `CLAUDE.md`, `agent-guide/*`, `README.md`)
2) 기존 코드베이스 패턴(에러 포맷/폴더 구조/네이밍/테스트)
3) 이 문서의 코딩 스타일 규칙

### MUST

- **범위 준수**: 요청 범위 밖 리포맷/리네이밍/리팩토링 금지 (필요하면 먼저 합의)
- **최소 Diff**: 불필요한 파일 이동/스타일 변경 금지
- **경계 검증**: 외부 입력(HTTP/env/파일/DB/LLM 출력)은 초기에 validate/normalize
  - ✗ `data = req.json(); process(data["key"])` (검증 없이 직접 사용)
  - ✓ `data = MySchema.model_validate(req.json()); process(data.key)`
- **에러 처리**: 예외를 삼키지 않기. Fail-fast 원칙
  - ✗ `except: pass`, `except Exception: return default_value`
  - ✗ `value = config.get("key") or "fallback"` (에러를 기본값으로 흡수)
  - ✓ 타입별 catch + 로그 + 사용자 메시지 분리. 폴백은 최상위 에러 경계에서만 허용
- **리소스 수명주기**: HTTP client/DB pool/모델/토크나이저는 요청마다 생성 금지, 앱 수명주기로 관리
- **설정 분리**: 하드코딩 대신 env/설정 파일. 안전한 기본값 + 명확한 실패

### SHOULD

- **명확성 우선**: early return, 의미 있는 네이밍, 단일 책임(SRP)
- **독립 작업 병렬화**: 불필요한 순차 await/N+1 피하기
- **접근법 검토**: 비사소한 로직 변경 시 코딩 전 더 단순한 접근법 검토. 여러 해석이 가능한 요청은 선택지를 나열하고 확인 후 진행

### AVOID

- **에러 뭉개기**: 검증 실패(400)/권한(401)/외부 장애(502) 구분 없이 "그냥 500" 금지
- **과도한 방어적 코딩**: 실패를 숨기는 패턴 금지
  - ✗ `result = api_call() ?? default_value` (실패가 버그인데 기본값으로 숨김)
  - ✗ `.catch(() => fallback)` (에러 원인 추적 불가)
  - ✓ 에러를 내서 원인을 추적하고 제대로 고치기. 폴백은 global error boundary에서만
- **느슨한 계약**: dict/str 기반 지양. API/LLM 출력은 Pydantic/Zod 등 스키마 사용

### 언어별

- **Python**: public 함수/서비스 경계에 타입 힌트. request/response/LLM 출력은 `pydantic`. async에서 블로킹 I/O 금지
- **TS/React**: public API/props/return 타입 명확히 (`any`/단언 지양). `Promise.all`+타임아웃/취소/재시도 정책 명시
- **SQL/DB**: 파라미터 바인딩. 클라이언트는 앱 수명주기로 관리

### 작업 후 기본 점검

작업 완료 후 아래 체크리스트로 검증한다:

1. **범위 검증**: 요청 범위를 벗어난 변경이 없는가?
2. **정합성**: 변경한 코드가 기존 패턴과 일관되는가?
3. **누락**: 요청된 사항 중 빠뜨린 것이 없는가?
4. **부작용**: 변경이 다른 코드에 영향을 미치는가?

검증 후 결과를 간결하게 요약한다 (변경점, 위치, 리스크).

## 코드 구조 규칙

> 코드 품질(에러 처리, 타입, 리소스)은 위 코딩 스타일 섹션이 담당.

### 우선순위 (충돌 시)

1) 기존 프로젝트의 파일 구조/패턴
2) 프로젝트 문서(`AGENTS.md`, `CLAUDE.md`, `agent-guide/*`, `README.md`)
3) 이 문서의 코드 구조 규칙

### MUST

- **기존 코드 파악 먼저**: 새 파일/함수 작성 전 (1) 기존 폴더 구조·네이밍 컨벤션 확인 (2) 유사 기능의 기존 유틸/함수 검색 → 있으면 재사용, 없으면 새로 작성
  - ✗ `monitor/app.py`가 이미 대시보드를 제공하는데, 확인 안 하고 `scripts/dashboard.py` 새로 작성
  - ✗ 프로젝트가 `lib/` 패턴인데 `helpers/` 디렉토리 생성
  - ✓ 검색으로 기존 유사 기능 확인 → 없음 확인 → 새로 작성
- **단일 역할 원칙**: 한 파일에 UI/비즈니스 로직/데이터 접근/타입/유틸 중 2개 이상 섞이면 분리. 무관한 유틸을 API 라우트 등에 추가 금지
- **의존성 단방향**: UI → 비즈니스 로직 → 데이터 접근 → 타입 방향 유지, 순환 의존 금지

### SHOULD

- **신규 기능은 구조 먼저 제시**: 파일 2개 이상 생성 예상 시, 파일 트리를 먼저 보여주고 사용자와 합의
- **Co-location**: 기능별로 관련 파일을 물리적으로 가까이 배치 (기존 프로젝트 패턴 우선)
- **내보내기 경계 명확히**: `index.ts`로 공개 API 명확히, 외부에서 내부 구현 파일 직접 접근 금지

### AVOID

- **만능 서랍/God 모듈**: `utils.ts`에 무관한 함수 누적 금지 → 기능별 분리
- **과도한 사전 추상화**: 구현체 1개뿐인데 인터페이스+팩토리+DI 만들지 않기. 2개 이상 필요해지면 그때 도입

## 보안 규칙

> 코드 품질(경계 검증, 설정 분리)은 위 코딩 스타일 섹션이 담당. 이 섹션은 보안 전용.

### 시크릿 관리

- 하드코딩 시크릿 **절대 금지**, 반드시 환경변수 사용
  - ✗ `API_KEY = "sk-abc123..."` (코드에 직접)
  - ✓ `API_KEY = os.environ["API_KEY"]` + 앱 시작 시 스키마 검증

### 취약점 방지

- **SQL 인젝션**: 문자열 연결 금지 → 파라미터 바인딩/ORM 사용
  - ✗ `f"SELECT * FROM users WHERE id = {user_id}"`
  - ✓ `cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))`
- **XSS**: `dangerouslySetInnerHTML` 사용 시 반드시 DOMPurify 적용
- **CSRF**: 상태 변경은 POST/PUT/DELETE + CSRF 토큰 또는 SameSite 쿠키

### 보안 문제 발견 시 대응

1. 현재 작업 중단
2. 심각도 평가: P0(즉시), P1(긴급), P2(중요)
3. OWASP Top 10 기반 점검
4. 수정 후 검증 + 코드베이스에서 유사 패턴 추가 검색

### 커밋 전 체크리스트

- [ ] 하드코딩된 시크릿 없음
- [ ] 모든 사용자 입력 검증됨
- [ ] 민감 데이터 로깅/에러 메시지 노출 없음

## 에이전트 운영 (Codex 절차형 대응)

Codex는 자동 에이전트 위임을 지원하지 않으므로, 아래 역할을 직접 수행한다.

### 역할별 트리거 및 절차

| 순위 | 역할 | 트리거 조건 | 절차 |
|------|------|------------|------|
| 1 | **빌드 해결** | 빌드 실패 (`npm run build`, `tsc`, `pnpm build` 에러) | 최우선 처리, 최소 변경으로 해결 |
| 2 | **보안 점검** | 보안 민감 코드 (auth/login/session/token, 암호화, 시크릿 처리) | OWASP Top 10 기반 점검 강화 |
| 3 | **계획 수립** | 복잡한 요청 (파일 3개+ 수정 예상, 아키텍처 결정 필요) | **계획 먼저 정리** → 사용자와 논의 후 확정 |
| 4 | **작업 점검** | 작업(코드/문서/리포트/계획서) 완료 후, 또는 "점검해줘/확인해줘" 요청 | 기본 점검 수행 |

일반 흐름: 계획 수립(초안) → 사용자 논의 → 구현 → 작업 점검 → (보안 시) 보안 점검 → (빌드 에러 시) 빌드 해결

### 직접 처리 조건 (역할 분리 불필요)

- 단순 질문/설명 요청
- 1-2줄 간단한 수정
- 사용자가 "직접 해줘" 명시적 요청
- 파일 1-2개만 수정하는 간단한 작업

### 역할별 스킬 연계

빠른 점검은 직접 수행하되, 심층 분석이 필요하면 해당 스킬을 읽고 따른다.

| 직접 수행 (빠른 점검) | 스킬 참조 (심층 분석) |
|----------------------|---------------------|
| 작업 점검: 기본 점검 | `code-review/SKILL.md`: 심층 리뷰 |
| 보안 점검: 자동 체크 | `code-simplifier/SKILL.md`: 코드 간소화 |
| 빌드 해결: 에러 수정 | `feedback-analysis/SKILL.md`: 피드백 분석 |
| 계획 수립: 계획 초안 | `update-docs/SKILL.md`: 문서 업데이트 |

### 복잡한 작업 착수 시 정리 프로토콜

복잡한 작업(파일 3개+ 수정) 시작 전 아래 항목을 명시적으로 정리한 후 진행한다.

| 항목 | 필수 | 설명 |
|------|:----:|------|
| **task** | O | 수행할 작업 (1문장) |
| **context_files** | O | 관련 파일 경로 목록 |
| **constraints** | O | 제약 조건 (변경 범위, API 호환성 등) |

작업 완료 후: 요청 범위를 벗어난 변경이 없는지 확인한다.

### 행동 규칙

- **파일 삭제 금지**: 파일/디렉토리 삭제는 반드시 사용자 확인 후 진행. 셸 명령(`rm`)뿐 아니라 내장 파일 도구를 통한 삭제도 동일. 리팩토링·정리 목적의 삭제도 사전 합의 필수
- **실패 시 재계획**: 구현 중 차단(의존성 충돌, 설계 오류) 발생 시 즉시 중단 → 재계획. 밀어붙이기 금지
- **버그 수정 자율성**: 버그 수정 요청 시 로그/에러/테스트를 직접 추적→수정. 단순 확인 질문 최소화. 범위 확대 시(파일 3개+) 합의
- **교훈 기록**: 사용자로부터 수정/피드백 받으면 `.codex/lessons.md`에 기록. 같은 유형의 에러 2회 이상 발생 시 기록 필수
- **계획 대비 확인**: 계획을 세운 경우, 구현 완료 후 계획 대비 누락 없는지 최종 확인한다

## 레퍼런스 검증 규칙

> 학술 논문·연구 보고서·기술 사양서를 인용하거나, 논문 수식을 코드로 구현하거나, benchmark 수치를 비교할 때 적용.
> 라이브러리/프레임워크 공식 문서는 대상이 아니다.

### MUST — 논문 인용

- **원문 확인 없이 구체적 내용 언급 금지**. "아마 ~일 것이다" 식의 추측 답변 금지
- **Abstract만으로 답변 금지**. 구체적 내용은 해당 섹션 본문까지 읽어야 한다
- **사용자 제공 자료 우선**. 웹 검색은 그 이후에 보충용
- **전문 확보 순서**: 사용자 자료 → ar5iv → 웹 검색 → Semantic Scholar → PubMed Central → 저자 홈페이지 → HTML 버전
- **접근 실패 시 솔직하게 고백**. 읽지 못한 부분을 추측으로 채우지 않는다

### MUST — 수식↔코드 대조

- **코드가 "Eq.(X) 구현"이라고 주장하면, 항 단위 대조표를 만들어 검증**
  - ✗ "added mass를 effective mass로 치환했으니 Eq.(21)과 같다" (개념 수준 판단)
  - ✓ 논문 Eq.(21) 우변: `F_h + (ρ_p-ρ_f)Vg`, 분모: `ρ_p V`. 코드: `F_total / mass_eff` where `mass_eff = mass_p + m_fluid` → 분모가 다르다. faithful 아님
- **코드 내부에 "heuristic", "exploratory", "project-specific" 라벨이 있으면 faithful로 전제하지 않기**

### MUST — benchmark 수치 비교

- **비교 전에 논문 원문에서 해당 무차원수의 정의 수식을 직접 확인**
  - ✗ Wang 2008 `Re_max = 484.75`를 그대로 우리 `Re_standard`와 비교 → "30% gap" 오진
  - ✓ Wang Eq.(43) 확인 → `Re = ρ_p U D / μ` (particle basis) → 우리 `Re_standard = ρ_f U D / μ`와 다름 → 환산 후 비교
- **정의가 다르면 환산식을 명시적으로 쓰고 나서 비교**

### MUST — 디버깅 사고

- **내가 구현한 코드는 "틀렸다"를 기본 전제로 시작. 맞다는 증거를 찾기**
- **가설은 확증이 아니라 반증을 먼저 설계**
- **수식 오류, 정의 오독 같은 단순 원인을 전부 배제한 뒤에만 "구조적 한계" 결론 허용**
- **새 실험 설계보다 기존 코드/수식 검증이 먼저**
- **같은 축에서 반복 실패하면 실험 멈추고 전제 목록 작성**

### 검증 프로토콜

```
1. 논문/레퍼런스 관련 질문 수신
2. 사용자가 파일/URL 제공했으면 해당 자료부터 읽기
3. 없으면 웹 검색 → ar5iv/HTML 확보 → 본문 읽기
4. 해당 섹션을 실제로 읽은 후에만 답변
5. 답변 시 출처 섹션 명시 (예: "Section 3.2에 따르면…")
```

### AVOID

- **기억 기반 인용**: 원문 확인 없이 "이 논문에서는 ~한 방법을 제안했습니다"
- **숫자/수치 날조**: 원문에서 확인한 수치만 언급
- **논문 간 혼동**: 유사한 제목/저자의 다른 논문 내용을 섞는 것
- **존재하지 않는 논문 인용**: 검색 결과로 존재가 확인된 논문만 인용

### 예외

- **일반 상식 수준**은 검증 불필요: "LBM은 Boltzmann 방정식 기반이다" 등
- 사용자가 **"검색 없이 아는 선에서"** 명시적 요청 시 — 단, "검증되지 않은 내용"임을 고지
